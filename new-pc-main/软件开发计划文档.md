## 一、项目概述

智慧图书馆座位预约系统是一套面向高校图书馆场景的座位管理解决方案，旨在解决图书馆座位资源利用率低、占座现象严重、管理效率低下等问题。系统采用多端架构设计，包含 PC 桌面端（Electron）、Web 端和 Android 移动端，覆盖学生预约座位、签到/签退、管理员管理座位等核心业务场景。

系统整体架构为前后端分离模式：

- **PC 桌面端 / Web 前端**：基于 Vue 3 + Vite 构建，通过 Electron 封装为桌面应用，使用 Element Plus 组件库提供现代化 UI 交互
- **Android 移动端**：包含用户端 App 和管理员端 App，使用 Java + Android SDK 原生开发，通过 Retrofit 与后端 API 通信
- **后端服务**：基于 Node.js + Express 5 构建 RESTful API，使用 JSON 文件存储数据，通过 JWT 实现身份认证

## 二、项目功能分析

### 2.1 功能模块划分

系统从用户角色维度可划分为两大子系统：**用户端**和**管理端**。每个子系统进一步划分为若干功能模块，具体如下：

```
智慧图书馆座位预约系统
├── 用户端
│   ├── 用户认证模块
│   ├── 座位浏览与预约模块
│   ├── 预约管理模块
│   ├── 签到/签退模块
│   ├── 个人中心模块
│   ├── 通知消息模块
│   ├── 收藏模块
│   └── 历史记录模块
└── 管理端
    ├── 数据统计模块
    ├── 座位管理模块
    ├── 预约管理模块
    └── 用户管理模块
```

### 2.2 各模块功能详述

#### 2.2.1 用户认证模块

| 功能项 | 描述 |
|-------|------|
| 用户注册 | 支持用户名、密码、姓名、学号注册，密码使用 bcrypt 加密存储 |
| 用户登录 | 支持用户名密码登录，返回 JWT Token（有效期7天） |
| 管理员登录 | 内置管理员账号，登录后角色标识为 admin |
| Token 验证 | 所有需认证接口通过 Bearer Token 进行身份校验 |
| 学号绑定 | 登录后可绑定学号信息 |

#### 2.2.2 座位浏览与预约模块

| 功能项 | 描述 |
|-------|------|
| 楼层座位概览 | 展示各楼层空余座位数量及使用率进度条 |
| 可视化座位地图 | 以图形化方式展示各楼层座位分布，区分可用/已占/不可用状态 |

| 时间冲突检测 | 自动检测用户已有预约是否存在时间冲突 |

## 四、过程模型选择

### 4.1 选择的过程模型

本项目选择 **增量模型（Incremental Model）** 作为软件开发过程模型。

### 4.2 选择理由

1. **多端并行开发需求**：系统包含 PC 桌面端、Web 端和 Android 移动端三个客户端，增量模型允许按端划分增量，各端可并行开发并逐步交付，避免瀑布模型中必须等待全部功能完成才能集成的弊端。

2. **核心功能优先交付**：图书馆座位预约系统的核心功能（登录注册、座位浏览、预约、签到签退）与辅助功能（收藏、通知、信用积分）优先级明确，增量模型可确保核心功能在最早期增量中交付，辅助功能在后续增量中补充。

3. **降低开发风险**：增量模型将项目分解为多个可管理的增量，每个增量完成后可进行评审和测试，及早发现问题，降低项目整体风险。相比瀑布模型，避免了"大爆炸"式集成带来的风险。

4. **适应需求变化**：作为课程项目，需求在开发过程中可能发生调整。增量模型在每个增量之间允许根据反馈调整后续增量的需求，比瀑布模型更具灵活性。

5. **早期可用原型**：第一个增量完成后即可得到可运行的系统原型，便于向指导老师展示进展和收集反馈。

6. **不适合纯敏捷的原因**：团队规模较小（4人），且为课程项目有明确的交付节点，完全敏捷的迭代节奏可能难以保证文档和交付物的完整性，增量模型在灵活性和规范性之间取得了更好的平衡。

### 4.3 模型应用方式

将项目划分为 **4个增量**：

| 增量 | 内容 | 交付物 |
|------|------|--------|
| 增量1 | 后端 API + 用户认证 + 座位浏览预约 + PC前端基础框架 | 可登录、浏览座位、预约座位的 PC 端应用 |
| 增量2 | 签到签退 + 预约管理 + 通知系统 + 个人中心 | 完整用户端 PC 应用 |
| 增量3 | Android 用户端 App + 收藏 + 历史记录 + 信用积分 | Android 移动端应用 |
| 增量4 | 管理端（PC + Android）+ 数据统计 + Electron 桌面封装 | 完整系统交付 |

每个增量内部采用小型瀑布流程：需求确认 → 设计 → 编码 → 测试 → 交付评审。

### 6.3 进度计划理由

1. **需求分析与设计前置**：将需求分析和系统设计安排在项目前期，确保后续开发有明确的技术方案和接口规范，避免开发过程中的返工。

2. **增量间合理重叠**：相邻增量之间存在1周左右的重叠期，因为后端 API 开发完成后前端即可并行开发，无需等待前一增量的全部工作完成。

3. **后端优先策略**：增量1中优先完成后端 API，为所有前端开发提供数据支撑。后端接口的尽早稳定可减少前端对接时的调整成本。

4. **PC 端优先于移动端**：PC 端（增量1、2）先于 Android 端（增量3）开发，因为 PC 端可同时覆盖 Web 和桌面两种使用场景，优先交付价值更大；同时 PC 端开发过程中积累的 API 对接经验可直接复用到 Android 端。

5. **管理端最后开发**：管理端（增量4）依赖用户端产生的数据（预约记录、用户信息等），安排在用户端基本完成后开发，便于使用真实数据进行测试。

6. **集成测试与部署预留充足时间**：多端系统集成涉及 PC 端与后端的通信、Android 端与后端的通信、Electron 封装等，集成测试预留1.5周确保质量。

## 七、风险管理

| 风险项 | 可能性 | 影响 | 应对措施 |
