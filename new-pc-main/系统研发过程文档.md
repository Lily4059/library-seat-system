**需求分析阶段组长：董占隆**

### 1.1 引言

#### 1.1.1 编写目的

本文档是智慧图书馆座位预约系统的需求分析文档，旨在明确系统的功能需求、非功能需求和业务约束，为后续的系统设计、实现和测试提供依据。本文档的预期读者包括：

- 项目开发团队成员
- 项目指导老师
- 图书馆管理人员（需求方代表）

#### 1.1.2 背景

**项目名称**：智慧图书馆座位预约系统

**项目提出方**：软件工程课程组

**项目开发方**：智慧图书馆座位预约系统研发小组

高校图书馆是学生学习的重要场所，但长期以来图书馆座位管理面临诸多问题。传统的人工管理方式无法有效监控座位使用状态，导致"占座"现象严重、座位资源利用率低、管理效率低下。随着信息化建设的推进，利用信息技术实现座位资源的智能化管理已成为高校图书馆建设的迫切需求。

当前图书馆座位管理存在的核心问题：

| 问题编号 | 问题描述 | 影响范围 |
|---------|---------|---------|
| P1 | 占座现象严重，物品占座但无人使用 | 全部楼层 |
| P2 | 学生无法提前了解座位空闲情况，到馆后寻找座位效率低 | 学生端 |
| P3 | 管理员无法实时掌握座位使用率和预约情况 | 管理端 |
| P4 | 缺乏有效的信用约束机制，爽约和占座成本极低 | 全系统 |
| P5 | 纸质登记方式效率低，数据难以统计和分析 | 管理端 |

### 1.2 任务概述

#### 1.2.1 目标

本项目旨在开发一套智慧图书馆座位预约系统，实现以下目标：

1. **在线预约**：学生可通过 PC 端或移动端在线预约座位，提前规划学习时间
2. **签到签退**：通过签到签退机制确保座位实际使用，杜绝占座现象
3. **信用约束**：建立信用积分体系，对爽约、占座等行为进行约束
4. **可视化管理**：提供可视化座位地图，直观展示座位状态
5. **数据统计**：为管理员提供座位使用率、预约统计等数据分析
6. **多端覆盖**：支持 PC 桌面端、Web 端和 Android 移动端

#### 1.2.2 用户的特点

| 用户类型 | 描述 | 计算机水平 | 使用频率 | 使用场景 |
|---------|------|-----------|---------|---------|
| 学生用户 | 高校在校学生，18-25岁 | 熟练使用手机App和浏览器 | 高（每日1-3次） | 课前预约、到馆签到、离馆签退 |
| 管理员 | 图书馆工作人员，25-50岁 | 基本操作电脑 | 中（每日1次） | 查看统计、管理座位、处理异常 |

#### 1.2.3 假定和约束

| 编号 | 假定/约束 | 说明 |
|------|----------|------|
| AC-01 | 网络环境 | 假设系统运行在校园局域网环境下，网络稳定 |
| AC-02 | 并发量 | 假设同时在线用户不超过500人，峰值并发不超过100 |
| AC-03 | 数据量 | 假设注册用户不超过10000人，座位数不超过1000 |
| AC-04 | 开发周期 | 项目开发周期为7周，采用增量模型 |
| AC-05 | 技术栈 | 前端 Vue 3 + 后端 Node.js + 移动端 Android Java |
| AC-06 | 数据存储 | 使用 JSON 文件存储，无需部署独立数据库服务 |
| AC-07 | 平台约束 | PC 端支持 Windows，移动端支持 Android 8.0+ |

### 1.3 用例分析

#### 1.3.1 系统Actor分析

系统存在两类 Actor：

**1. 学生用户（Student）**

- 描述：图书馆的普通使用者，通过系统进行座位预约、签到签退等操作
- 访问方式：PC 桌面端 / Web 浏览器 / Android App
- 权限范围：座位浏览、预约、签到签退、个人信息管理、收藏、通知查看

**2. 管理员（Admin）**

- 描述：图书馆管理人员，通过系统管理座位、查看统计数据
- 访问方式：PC 桌面端 / Android App
- 权限范围：座位管理、预约管理、用户管理、数据统计

### 1.4 对象-关系建模

系统核心实体及其关系如下：

![类图](uml/03_类图.puml)

**核心实体说明：**

| 实体 | 属性 | 说明 |
|------|------|------|
| User | id, username, password, name, studentNo, creditScore, totalStudyTime, phone, avatar, createdAt | 用户实体，密码使用 bcrypt 加密存储 |
| Seat | id, seatNo, floor, area, status, window, power, silent | 座位实体，status 包含 available/booked/unavailable |
| Reservation | id, userId, seatId, seatNo, seatType, startTime, endTime, status, checkInTime, checkOutTime, cancelReason, createdAt | 预约实体，status 包含 pending/checked_in/completed/cancelled/expired |
| Notification | id, userId, type, title, summary, content, actionLink, actionText, isRead, createdAt | 通知实体，type 包含 booking/system/points |
| Favorite | id, userId, itemType, itemId, createdAt | 收藏实体，itemType 包含 seat/room |
| LeisureRoom | id, name, openTime, peopleCount | 自习室实体 |

**实体关系：**

- User 与 Reservation：一对多（一个用户可有多个预约）
- User 与 Notification：一对多（一个用户可有多条通知）
- User 与 Favorite：一对多（一个用户可有多个收藏）
- Seat 与 Reservation：一对多（一个座位可被多次预约）
- Favorite 与 Seat：多对一（多个收藏可引用同一座位）
- Favorite 与 LeisureRoom：多对一（多个收藏可引用同一自习室）

1. **注册/登录流程**：用户打开系统 → 未登录则跳转登录页 → 输入用户名密码 → 系统验证 → 返回 Token → 存储到本地 → 进入首页

2. **座位预约流程**：进入首页 → 查看楼层概览 → 选择楼层 → 查看座位地图 → 选择座位 → 设定时间段 → 系统校验（开馆时间、时间冲突、最短时长） → 确认预约 → 推送通知

3. **签到签退流程**：进入我的预约 → 点击签到 → 系统记录签到时间 → 学习结束 → 点击签退 → 系统计算学习时长 → 累加至用户总时长 → 推送通知

4. **通知查看流程**：进入通知中心 → 查看未读/全部通知 → 标记已读/删除 → 返回

5. **收藏管理流程**：在座位地图或自习室列表 → 点击收藏 → 在收藏页管理 → 取消收藏/批量删除

**管理端核心业务流程：**

1. **数据统计**：管理员登录 → 进入管理端 → 查看座位/用户/预约统计

2. **座位管理**：进入座位管理 → 查看座位列表 → 单个/批量修改状态 → 保存

3. **预约管理**：进入预约管理 → 查看所有预约 → 删除异常预约

4. **用户管理**：进入用户管理 → 查看用户列表 → 查看用户详情

#### 2.2.2 用户认证模块设计

**功能描述**：实现用户的注册、登录和身份验证功能。

**设计要点**：
- 密码使用 bcrypt 算法加密存储（salt rounds = 10）
- 登录成功后生成 JWT Token（有效期 7 天）
- 管理员账号内置（username: admin, password: admin123456）
- 所有需认证的接口通过 `authMiddleware` 中间件校验 Token
- Token 从请求头 `Authorization: Bearer <token>` 中提取

**状态管理**：
- 前端使用 Pinia 的 `userStore` 管理 Token 和用户信息
- Token 存储在 localStorage 中，页面刷新后自动恢复登录状态
- 路由守卫检查 `requiresAuth` 元信息，未登录重定向到登录页

#### 2.2.3 座位浏览与预约模块设计

**功能描述**：实现座位浏览、筛选和预约功能。

**设计要点**：
- 座位数据按楼层组织（1F-4F），每层包含多个区域（A/B/C/D区）
- 座位属性包含标签信息：靠窗（window）、电源（power）、静音区（silent）
- 座位状态通过 `statusOverrides` 覆盖机制实现管理员修改
- 预约创建时进行多重校验：参数完整性、时间格式、时间段合法性、闭馆时间、最短时长、时间冲突
- 时间冲突检测使用区间重叠算法：`isOverlap(aStart, aEnd, bStart, bEnd)`

**预约校验规则**：

| 校验项 | 规则 | 错误码 |
|--------|------|--------|
| 参数完整性 | seatId、seatNo、startTime、endTime 不能为空 | 400 |
| 时间格式 | 必须为合法的 ISO 日期格式 | 400 |
| 时间段合法性 | 结束时间必须晚于开始时间 | 400 |
| 闭馆时间 | 不可跨天，时间范围在 08:00-22:00 内 | 400 |
| 最短时长 | 预约时长不少于 30 分钟 | 400 |
| 时间冲突 | 与用户已有活跃预约不可重叠 | 409 |

#### 2.2.4 预约管理模块设计

**功能描述**：实现预约的查看、取消和状态流转。

**预约状态机**：

```
pending（待签到）──签到──→ checked_in（使用中）──签退──→ completed（已完成）
     │                                             
     └──取消──→ cancelled（已取消）                  
```

**设计要点**：
- 取消预约需填写取消原因
- 只有 pending 和 checked_in 状态的预约可取消
- 预约列表按开始时间倒序排列

#### 2.2.5 签到签退模块设计

**功能描述**：实现座位使用的签到和签退操作。

**设计要点**：
- 签到：验证预约状态为 pending → 记录签到时间 → 状态变更为 checked_in → 推送通知
- 签退：验证预约状态为 checked_in → 记录签退时间 → 计算学习时长（分钟） → 累加至用户 totalStudyTime → 状态变更为 completed → 推送通知
- 学习时长计算：`studyTime = Math.round((checkOutTime - checkInTime) / 60000)`

#### 2.2.6 通知消息模块设计

**功能描述**：实现通知的查询、标记已读和删除功能。

**设计要点**：
- 通知类型：booking（预约相关）、system（系统通知）、points（积分变动）
- 支持分页查询（page、pageSize 参数）
- 支持按类型筛选
- 未读计数实时查询
- 自动通知：预约成功、签到成功、签退成功时由后端自动创建

#### 2.2.7 收藏模块设计

**功能描述**：实现座位和自习室的收藏管理。

**设计要点**：
- 收藏类型：seat（座位）、room（自习室）
- 防重复收藏：添加收藏前检查是否已存在
- 收藏列表关联查询：收藏记录关联座位和自习室详情
- 批量删除：支持一次删除多个收藏记录

#### 2.2.8 管理端模块设计

**功能描述**：实现管理员对座位、预约和用户的管理功能。

**设计要点**：
- 权限控制：通过 `authMiddleware` + `adminMiddleware` 双重中间件校验
- 座位状态覆盖：管理员修改座位状态通过 `statusOverrides` 机制，不直接修改原始数据
- 批量操作：支持批量修改座位状态
- 预约管理：查看所有预约（关联用户信息），可删除异常预约
- 数据统计：座位统计、用户统计、预约统计（含今日预约数）

### 3.2 系统通用类实现

#### 3.2.1 数据库通用类实现

**JsonStore 类（server/db/jsonStore.js）**

JsonStore 是系统的核心数据存储引擎，基于 JSON 文件实现数据持久化。每个 JsonStore 实例对应一个 JSON 文件，提供完整的 CRUD 操作。

```javascript
class JsonStore {
  constructor(filename) {
    this.filePath = join(dataDir, filename)
    this.cache = null
    this.load()
  }

  load() {
    if (existsSync(this.filePath)) {
      const content = readFileSync(this.filePath, 'utf-8')
      this.cache = JSON.parse(content)
    } else {
      this.cache = {}
    }
  }

  save() {
    writeFileSync(this.filePath, JSON.stringify(this.cache, null, 2), 'utf-8')
  }

  insert(collectionName, item) {
    const collection = this.getCollection(collectionName)
    const newItem = {
      ...item,
      id: item.id || this.cache['nextId'],
      createdAt: item.createdAt || new Date().toISOString()
    }
    collection.push(newItem)
    this.cache['nextId']++
    this.save()
    return newItem
  }
}
```

**核心方法实现说明**：

| 方法 | 实现要点 |
|------|---------|
| load() | 启动时从文件读取 JSON 数据到内存缓存，文件不存在则初始化空对象 |
| save() | 每次写操作后立即将缓存写入文件，确保数据持久化 |
| insert() | 自动生成自增 ID 和 createdAt 时间戳 |
| update() | 使用对象展开符合并更新字段 |
| findMany() | 支持函数谓词和对象谓词两种查询方式 |
| deleteMany() | 使用 filter 过滤后替换原集合 |

#### 3.2.2 其它通用类实现

**认证中间件（server/middleware/auth.js）**

```javascript
import jwt from 'jsonwebtoken'

const JWT_SECRET = 'library-seat-secret-key-2024'

export const authMiddleware = (req, res, next) => {
  const authHeader = req.headers.authorization
  if (!authHeader || !authHeader.startsWith('Bearer ')) {
    return res.status(401).json({ code: 401, message: '未登录，请先登录' })
  }
  const token = authHeader.split(' ')[1]
  try {
    const decoded = jwt.verify(token, JWT_SECRET)
    req.userId = decoded.userId
    req.username = decoded.username
    next()
  } catch (error) {
    return res.status(401).json({ code: 401, message: 'token已过期，请重新登录' })
  }
}

export const generateToken = (userId, username) => {
  return jwt.sign({ userId, username }, JWT_SECRET, { expiresIn: '7d' })
}
```

### 3.9 Electron桌面封装实现

Electron 主进程（electron/main.cjs）负责以下工作：

1. **创建窗口**：配置窗口尺寸（1400×900）、最小尺寸（1000×700）、安全策略
2. **启动后端服务**：通过 `child_process.spawn` 启动 Node.js 后端服务
3. **加载前端页面**：开发环境加载 `localhost:5173`，打包后加载 `dist/index.html`
4. **菜单配置**：配置 File、View、Help 菜单
5. **生命周期管理**：窗口关闭时停止后端服务

```javascript
function startServer() {
  const serverPath = app.isPackaged
    ? path.join(process.resourcesPath, 'server/index.js')
    : path.join(__dirname, '../server/index.js')
  serverProcess = spawn('node', [serverPath], {
    cwd: serverDir,
    env: { ...process.env, PORT: '3000' }
  })
}
```

打包配置将后端服务代码作为额外资源打包：

```json
{
  "extraResources": [
    { "from": "server", "to": "server" }
  ]
}
```

#### 4.2.4 系统通用类调试过程

在 JsonStore 的 `deleteMany` 方法实现中，最初使用 splice 在循环中删除元素，导致索引错位，部分记录未被删除。后改为使用 filter 过滤后替换原集合的方式解决：

```javascript
deleteMany(collectionName, predicate) {
  const collection = this.getCollection(collectionName)
  const initialLength = collection.length
  const filtered = collection.filter(item => !predicate(item))
  this.cache[collectionName] = filtered
  this.save()
  return initialLength - filtered.length
}
```

```javascript
const isOverlap = (aStart, aEnd, bStart, bEnd) => {

}
```
